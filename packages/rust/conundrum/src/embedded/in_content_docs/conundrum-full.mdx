When I first built Fluster I was using a third-party package to parse markdown and jsx, but I've since implemented about 80% of the parser that Fluster needs independent of any third party package. While this is largely a technical implementation detail, this means that Fluster can build a purpose driven, mobile friendly DSL (domain specific language) specifically for note taking.

<Admonition title="For Developers" folded foldable>

You can find the repository at [FlusterIO](https://github.com/flusterIO) -> **FlusterIPad** (poorly named monorepo) -> `packages` -> `rust` -> `conundrum`.

The language works like this:

It doesn't compile to some low level target like Assembly. Instead it compiles to a cross-language friendly object (JSON & Buffer serializable) that contains all of the data common to many note taking and blogging applications, and executes using a Rust interpreter capable of executing in a completely cross-platform & mobile friendly environment; a feature I was desperate for when I was living in my car and working on this model. This object contains keys for things like `outgoing_note_link_ids` that collects the id's from the `[My note link by id](note:myNoteById)` syntax that Fluster users, `citations` collected using the `[[cite:myCitation]]` syntax, `tags`, etc... and if the front matter is present, things like a `topic`, and a `subject`.

Along with this object there is a primary `content` property that is compiled based on a set of flags provided to the Conundrum compiler, things like: 

| Flags | Description |
| ----- | ----------- |
| `.preferMarkdownSyntax` | For now just removes all jsx components to allow the note to be rendered in markdown based applications, but will soon come with a special Markdown renderer for most internal components so that things like the title of an admonition isn't lost. |
| `.forAIInput` | Similar to `.preferMarkdownSyntax`, but this is set specifically when the output is meant to be consumed by AI. |
| `.preferPlainText` | Output the note to plain text. Useful for things like full-text search without the markdown and jsx syntax getting in the way. |

This type of compilation allows Fluster to compile a subset of the note to markdown (Swift can't run mdx without a webview) and provide the entire note to the full-text search with much improved reliability while still rendering the complete UI. Even a syntax based search function that prioritizes content found within titles, even on the jsx level can be built right into the language with this multi-language parser-combinator approach.

Think of the `[My note by id](note:myNoteId)` syntax Fluster uses. This syntax is Fluster specific, locking users into using Fluster for the remainder of their notes. If we instead abstract away the collection of this data into a shared syntax, we can leave the _insertion_ of the `note.id` field completely up to the application... all that matters is that the note has the _concept_ of an **id**. The same goes for the `jsx`. We don't have to use javascript for computation at all! Fluster intentionally left out the components like plots that might accept a function as a property until the end so that our own DSL can be implemented on _top_ of jsx. There might be room for some interoperability, but in all honesty I just think the general `jsx` syntax is a really decent syntax for content like notes that might contain a lot of nested children, but I think we can do better than Javascript for users new to code for things like math, while the code blocks inherent to markdown leave room for users that need access to a full programming language by making them executable. Things like python's operator casting to simplify math on vectors and Swift's simple `.myEnumVariant` syntax can all be a first step to something like Python's dataclasses for the user that might be interested in some more advanced usage of their devices without the need to learn an entire programming language.

The goal for this language is this:

1. Provide a _super_ high level programming layer to markdown. Add the ability to store variables in state, pass that state to the jsx components, and even create new jsx components that contain inputs that modify that state. Something like the `const $myState = false;` in a `conundrum` code block, and then `<Admonition foldable folded={$myState} >` to utilize that state.
2. Provide a high level wrapper for many of the most common mathematical and scientific routines used in applications typically written in languages like Python, while exposing the full capabilities of those languages at compile time with a shared memory layer. This would look like a `Number` or `Array` type that includes many of the common `numpy`/`scipy` methods built-in for the Conundrum language itself, but a sandboxed environment for each language that is supported available at compile time. Any plot output or other UI can be left to the application's implementation, as the data is returned as part of the compiled object.
3. Provide a real general-user first API as most of the users of this language will be people that never become professional developers. Things like a first-party AI module that can integrate directly with the rendered markdown, kind of breaking this language into two pieces: The markdown based syntax, and the logic that goes inside of a code block. I think the perfect language for note taking, blogging and other information-driven writing is the language that runs in both environments. Perhaps this might best be demonstrated with a possible future syntax, `!![My princess](ai:Show my pictures of layla)` which would return a vector of image data representing my sweet puppy (formatted as base64 or as a uint8 array depending on a provided flag). This could then be displayed by us, the developers as a carousel, a grid, or whatever UI you are trying to implement... the syntax would be the same and work _across_ applications, allowing users to work in this happy middle ground between the simplicity of Markdown and the complexity of something like Typescript and HTML.
4. With modern FFI and cross-language serialization libraries, there's no reason we can't have every nested language contribute to the parent language's memory layer. Exposing API's in the various languages should allow that, as long as those languages are available at compile time, that each API can serialize a subset of that language into a shared format that can be appended to the shared, Rust based memory layer. This would allow for things like calculating a complex array of numbers in Rust or C in 1 code block, and then building the plot directly from that data in Python in the code block below it. We can even implement an _index_ property to get around the linear requirement inherent in most languages.

While Fluster remains my own personal project for the time being, the compiler for this language that I once called Conundrum and now I'm too lazy to rename all of the files is completely open source. Markdown is great, but it's too simple for modern workflows. HTML and Typescript are even better, but they take years to learn well. There is a huge middle ground where users need the simplicity and quickness of a markdown based syntax with support for features that only something like mdx can offer, and while I'm happy to be the app that fills that middle ground, I don't want users to be forced to use my application just because their notes used that syntax at one point, which is why I'm encouraging other developers to consider building an application around the Conundrum compiler.
