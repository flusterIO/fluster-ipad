## Intro to Conundrum



If you've never written a line of code, don't worry, you don't have to learn much. _Writing_ your own functions and combining them to make an entire application can be quite complicated, but simply _calling_ functions is just as simple as mardown. In the end, the `<MyComponent>...</MyComponent>` syntax is just you _calling_ a function. All you need to know is the name of the component, and what properties that components accepts.


**Documented Properties**

- Strings
- Booleans
- Numbers
- Arrays (lists)
- Objects



**Not Documented Yet**
- Functions
  - This is my main priority after the initial release of Fluster, integrating a complete scripting language that allows users to pass functions to plot components. This feature was present in the previous release of Fluster, but due to the migration away from Javascript it has not yet been reimplemented.


## Components

Components are the core of jsx, and a fundamental reason of why the language is so popular. Jsx allows us to organize 'code' in a format that makes it incredibly easy to read, very unlike the traditional code that makes up the jsx components behind the scenes.

Fluster provides a growing list of components for you, and with snippets to help with each component, you'll rarely have to write a component by hand. That doesn't mean however that it's not beneficial to know how to edit them for the inevitable moment when you delete the wrong quotation mark or bracket. If you've written any programming language, or even if you haven't, this is _not_ as big of an ask as it my seem. Let's get started.

### The Basics

> Remember: Each component includes a set of short and long documentation that can be accessed using the `?` or `??` syntax, snippets to help with auto-completion, syntax highlighting within the editor, and some hopefully helpful error messages to help when you provide an incorrect property.

Here are the core pieces of information you need to know as it pertains to calling a jsx function.


- Components will always start with a `<` and a capital letter.
- Components can accept any number of properties. While components traditionally each have their own set of properties, Fluster bundles them into common, repeated models for familiarity, while some components still accept unique properties.
- `Children` is a unique property, being wrapped _between_ the component. This is a big part of what makes jsx so readable, as we insert regular mdx content _inside_ of the component where that content will actually render.
  - If a component **does** accept _children_, it's opening tag is closed with a single `>`, if it does not accept children the component accepts a shorthand where the second tag is not needed at all, and the first tag can be closed with a `/>`.


#### Examples

This `Card` component accepts `children`, so we use the `>` closing bracket on the opening tag.

```tsx -- title="components.mdx"
<Card title="My title" desc="My optional description">
This is my card's body.
</Card>
```

This `Toc` (table of contents) component does not accept any children, so we don't need the closing tag and we can write it as:

```tsx -- title="components.mdx"
<Toc />
```

### Strings
> See [these](https://developer.mozilla.org/en-US/docs/Learn_web_development/Core/Scripting/Strings) docs for a much more thorough explanation

1. A `string` is just a fancy word for words and letters. Numbers can also be strings, which might be confusing, but Conundrum will catch that for you if you make that mistake.
2. To create a string, just wrap the content in either single `'`, double `"`, or a backtick "`" (top left of the keyboard.).
3. Conundrum currently does not support line breaks inside of strings. This might sound paradoxical to experienced programmers, but remember... pretty much everywhere a string is accepted, more Conundrum content is accepted as well.
4. Wrapping your string in `{}`, specifically when applying it to a component is optional.

#### Examples

```tsx -- title="strings.mdx"
<Card
    // I can wrap the title with single quotes, 
    // double quotes or backticks.
    title='My title'
    // But because this has an apostrophe, I can't
    // use the single quotes here without escaping it,
    // So I'll use backticks here.
    desc={`This is my description's content.`}
>
...
```


#### _Experimental_ String escaping
> This will be fully supported in a future update. It may or may not work currently...

If we _really_ wanted to use single quotes inside of another pair of single quotes, or double quotes inside of double quotes, we can do that as:

```tsx -- title="strings.mdx"
<Card 
    // If I really need to include matching quotations within the string itself,
    // I have to lead it with a leading `\` character.
    desc={"This is my \"quotation\" here"}
/>
```

## Booleans


Booleans are just true or false values, indicated by `true` or `false` in lowercase. To make things easier, jsx allows a shorthand syntax where when a property is `false` by default, it can be set true by simply passing the property's name.

This gives a situation where in the following code, `shrink` and `important` will both evaluate to true, and some other value, say `notImportant` will evaluate to false just by omission.


```tsx -- title="booleans.mdx"
<Card 
    shrink={true}
    important
>
...
```

## Numbers

Numbers in javascript are straight-forward, defined exactly as you'd expect. One important property to note however, is that to provide a number as a value to a jsx property, you _must_ wrap the value in curly brackets. This gives something that looks like:

```tsx
<Grid medium={2} >
...
```

Note here that 2, and 2.0 are actually _not_ the same thing. Conundrum, like many programming languages, makes a distinction between integers and floats (decimals).


## Arrays

Arrays are just lists and are familiar to every programmer, and thankfully, they're incredibly easy to work with in conundrum. Like numbers, the value must be wrapped in a set of `{}` brackets, but inside of those brackets, the actual `Array` notation requires another set of **square** brackets. This ends up looking like the following:

```tsx -- title="arrays.mdx"
// This is an empty array. 
// This pie plot is sad because it has no data, but the syntax is valid.
<PiePlot data={[]} />
```

To include values within the array, just seperate the normal conundrum syntax (not necessarily jsx syntax, so no extra curly brackets everywhere) of whatever data type you're trying to insert with commas.

```tsx -- title="arrays.mdx"
<MyFakeComponent 
    myArrayOfNumbers={[1, 2, 3, 4, 5]}
    myArrayOfStrings={["One", "two", "three", "four", "five"]}
    myArrayOfObjects={[
        {
            body: "My body",
            value: 2
        },
        {
            body: "My other body",
            value: 3
        }
    ]}
/>
```


What is much more likely however, is that whatever arrays you do work with inside of Fluster will be generated for you. There is currently a math extension that is in beta that will provide a very [numpy](https://numpy.org/doc/stable/) like array interface for plotting and mathematical calculations directly within your note.

## Objects

'Objects' are at the core of most programming languages, and javascript is no exception. This subject does however start to bridge the gap between what is required to use Fluster and what is the beginning of a software career, but I would encourage more advanced users to learn more [here](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Guide/Working_with_objects). Objects are defined as a set of key-value pairs, where the value can be almost anything, including another object.

```tsx
<MyFakeComponent 
    // Notice that there are 2 `{{}}` here.
    // 1 is required for jsx, the other for the actual object.
    // Confusing... I know.
    myObjectProperty={{
        name: "My name",
        value: 2,
        address: {
            street: "1 Main Street",
            city: "North Pole"
        }
    }}
/>
```
